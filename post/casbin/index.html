测试编码

package main
import (
	"fmt"
	"github.com/casbin/casbin/v2"
	gormadapter "github.com/casbin/gorm-adapter/v3"
	_ "github.com/go-sql-driver/mysql"
)
func main() {
    // 添加参数为true会自动创建casbin_rule表
	a, _ := gormadapter.NewAdapter("mysql", "root:root@tcp(192.168.1.115:3306)/casbin", true)
	e, _ := casbin.NewEnforcer("D:\\code\\go_code\\go_zero_source_code\\http\\demo\\casbin\\model.conf", a)
	sub := "alice1" // 想要访问资源的用户。
	obj := "data1"  // 将被访问的资源。
	act := "read"   // 用户对资源执行的操作。
	ok, err := e.Enforce(sub, obj, act)

	if err != nil {
		// 处理err/
	}
	if ok == true {
		// 允许alice读取data1
		fmt.Printf("通过")
	} else {
		// 拒绝请求，抛出异常
		fmt.Printf("不通过")
	}
}

权限控制模型#

ACL#

ACL（access control list） 访问控制列表。

最早的权限管理管理模型，原理核心:每一项资源，都有一个资源列表，通过资源列表来判断用户对资源的控制权限 。

ACL通过资源列表就能实现访问控制，非常简单，但其要维护大量的访问权限列表，性能上有缺陷，且当用户量巨大的时候访问列表的维护工作比较繁重。

RBAC#

（Role Based Access Control）基于角色的访问控制

在ACL模型的基础上添加了角色概念，通过用户和角色关联，角色和权限关联做分离。减少权限数据的频繁变更。

RBAC的三种模型

  • RBAC0 基础模型用户和角色多对多，角色和权限多对多
  • RBAC1 基于RPAC0模型引入了角色继承关系，角色有了上下级关系
  • RBAC2 基于前两种模型上进行了角色访问控制

ABAC#

ABAC(Attribute bases access control)基于属性的权限验证

更细粒度的权限控制(权限系统设计的未来)

ABAC基于==策略==进行访问控制，ABAC通过动态计算一个或一组属性来判断是否满足某个条件来进行授权判断。一般配合RBAC使用. 属性通常分为四类

  • 用户属性(年龄 性别，职位)
  • 环境属性(时间，内外网)
  • 操作属性(增、删，改，查)
  • 对象属性(文章，也称资源属性)

模型选择#

取决于组织规模大小(用户数量，业务大小)，ABAC设计比较复杂，通常在一般系统使用RBAC来替换ABAC。

**ps：**当满足一下条件就需要使用ABAC

  • 拥有大量用户
  • 精准细粒度的控制权限
  • 确保隐私和安全
